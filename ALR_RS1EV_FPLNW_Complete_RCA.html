1. Executive Summary

Root Cause Identified: The INCOMPINTVS validation check in validations.py incorrectly treats expected EV charger removal as a data quality failure, causing premises to be excluded from the study.
Aspect Finding
Issue Type Validation Logic Error
Affected Study FPLC-RS1EV (FPLNW EV Customers)
Root Cause INCOMPINTVS check doesn't account for expected EV charger mid-cycle removal
Impact Customer count dropped from 101 to 27 (73% reduction)
Fix Location validations.py lines 1548-1557
Fix Complexity Low - Add filter to exclude EV meters from INCOMPINTVS check

4. Root Cause Analysis

System Architecture

Data Flow:
  1. Data Fetch (gulf_daily.json) - Fetches meter and interval data from source databases
  2. Validation (validations.py) - Checks data quality, generates errors
  3. Study Population - Excludes bad_studypoints from active_premises

The Problematic Code

File: validations.py
Lines: 1548-1557

# Current Code - Incomplete Intervals Check df2_grouped1 = df2_grouped[~df2_grouped.meterstop_dt.isna()].copy() error_df_append = df2_grouped1[df2_grouped1.meterstop_dt < df2_grouped1.billstop][ ['studyid', 'premiseid', 'rltv_mo', 'studypoint', 'meterstop_dt', 'billstop']]

How INCOMPINTVS errors cause exclusion (line 706):

bad_studypoints = bad_studypoints.append(error[error.errorcode == 'INCOMPINTVS'][['studyid', 'studypoint']])

How bad_studypoints are filtered out (lines 944-947):

# Filter points for not in badstudypoints i1 = points.set_index(['studyid', 'studypoint']).index i2 = bad_studypoints.set_index(['studyid', 'studypoint']).index points = points[~i1.isin(i2)].reset_index()

The Logic Flow

What happens when an EV charger is removed mid-billing cycle:
  1. EV charger physically removed on April 15
  2. Interval data correctly stops on April 15 (meterstop_dt = 2025-04-15)
  3. Billing period continues until April 28 (billstop = 2025-04-28)
  4. Validation check: meterstop_dt < billstop → TRUE
  5. INCOMPINTVS error generated
  6. Premise added to bad_studypoints
  7. Premise excluded from active study
  8. Customer count drops

Why This Is Wrong

EV chargers (channel 1000) are different from billing meters:

  • EV chargers can be physically removed mid-billing cycle - this is expected behavior
  • The billing meter (channel 1) continues independently
  • The current validation treats this expected behavior as a data quality failure

6. Recommended Solution

Solution Overview

Fix: Modify the INCOMPINTVS check to skip EV meters (channel 1000, metertype evmeter/uevmeter). EV chargers stopping mid-cycle is expected behavior, not a data quality issue.

Code Change

File: validations.py
Lines: 1548-1557

Current Code (Problematic)

# Incomplete Intervals Check print("Performing Incomplete Intervals Check") # Write to error table where the max meterstop_dt is less than the billstop df2_grouped1 = df2_grouped[~df2_grouped.meterstop_dt.isna()].copy() error_df_append = df2_grouped1[df2_grouped1.meterstop_dt < df2_grouped1.billstop][ ['studyid', 'premiseid', 'rltv_mo', 'studypoint', 'meterstop_dt', 'billstop']]

Updated Code (Fixed)

# Incomplete Intervals Check print("Performing Incomplete Intervals Check") # Write to error table where the max meterstop_dt is less than the billstop # EXCEPTION: Skip this check for EV meters (channel 1000) - expected mid-cycle removal df2_grouped1 = df2_grouped[~df2_grouped.meterstop_dt.isna()].copy() # Filter out EV meters from INCOMPINTVS check # EV chargers can legitimately stop mid-billing cycle when physically removed non_ev_data = df2_grouped1[ ~((df2_grouped1.channel == 1000) & (df2_grouped1.metertype.isin(['evmeter', 'uevmeter']))) ] error_df_append = non_ev_data[non_ev_data.meterstop_dt < non_ev_data.billstop][ ['studyid', 'premiseid', 'rltv_mo', 'studypoint', 'meterstop_dt', 'billstop']]

What This Fix Does

Scenario Before Fix After Fix
Regular meter stops before bill end INCOMPINTVS error INCOMPINTVS error (unchanged)
EV meter (ch 1000) stops before bill end INCOMPINTVS error → Excluded No error → Stays in study
Customer count Drops when EV chargers removed Stable - expected behavior allowed

7. Verification Queries & Steps

Pre-Implementation Verification

Query 1: Count current INCOMPINTVS errors for RS1EV

-- Run this BEFORE implementing the fix SELECT studyid, COUNT(*) as incompintvs_count FROM clr.error WHERE errorcode = 'INCOMPINTVS' AND studyid = 'FPLC-RS1EV' GROUP BY studyid; -- Expected result: ~63 errors (based on current data)

Query 2: Identify affected premises

-- List all premises with INCOMPINTVS errors in RS1EV SELECT DISTINCT premiseid, rltv_mo, errormessage FROM clr.error WHERE errorcode = 'INCOMPINTVS' AND studyid = 'FPLC-RS1EV' ORDER BY premiseid, rltv_mo;

Query 3: Verify these are EV meter errors

-- Confirm INCOMPINTVS errors are related to evmeter SELECT premiseid, errormessage, CASE WHEN errormessage LIKE '%evmeter%' THEN 'EV Meter' ELSE 'Other' END as meter_type FROM clr.error WHERE errorcode = 'INCOMPINTVS' AND studyid = 'FPLC-RS1EV';

Query 4: Check current active premise count

-- Current active premises in RS1EV SELECT studyid, rltv_mo, COUNT(DISTINCT premiseid) as active_premise_count FROM clr.active_premise WHERE studyid = 'FPLC-RS1EV' GROUP BY studyid, rltv_mo ORDER BY rltv_mo DESC LIMIT 12;

Post-Implementation Verification

Query 5: Verify no new INCOMPINTVS for EV meters

-- Run this AFTER implementing the fix (on new data) -- Should return 0 rows for EV meters SELECT premiseid, rltv_mo, errormessage FROM clr.error WHERE errorcode = 'INCOMPINTVS' AND studyid = 'FPLC-RS1EV' AND errormessage LIKE '%evmeter%' AND process_run_date >= '20260201'; -- Adjust date after implementation

Query 6: Monitor customer count recovery

-- Track active premise count over time SELECT rltv_mo, COUNT(DISTINCT premiseid) as customer_count, LAG(COUNT(DISTINCT premiseid)) OVER (ORDER BY rltv_mo) as prev_month_count, COUNT(DISTINCT premiseid) - LAG(COUNT(DISTINCT premiseid)) OVER (ORDER BY rltv_mo) as change FROM clr.active_premise WHERE studyid = 'FPLC-RS1EV' GROUP BY rltv_mo ORDER BY rltv_mo;

Query 7: Verify EV data is preserved up to meter stop

-- Check that EV interval data exists up to meter stop date SELECT fr.premiseid, fr.meterid, fr.channel, MIN(fr.read_strt_time) as first_reading, MAX(fr.read_strt_time) as last_reading, m.meterstop FROM clr.fetch_results fr JOIN clr.meters m ON fr.premiseid = m.premiseid AND fr.meterid = m.meterid WHERE fr.studyid = 'FPLC-RS1EV' AND fr.channel = 1000 AND fr.premiseid IN (70002132, 70002972, 70005262) GROUP BY fr.premiseid, fr.meterid, fr.channel, m.meterstop;

Conclusion

Summary: Both issues (customer count drop and data stopping after meter change) are caused by the same root cause - the INCOMPINTVS validation incorrectly penalizes expected EV charger removal.

Fix: Skip INCOMPINTVS check for channel 1000 (evmeter/uevmeter).

Expected Outcome: Customer count will stabilize and EV interval data will be preserved up to the meter stop date.
